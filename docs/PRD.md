# 个人知识库笔记软件 PRD v0.1

## 1. 产品概述

### 1.1 产品定位
一款面向个人使用的跨端笔记与知识库软件，覆盖 macOS、iPhone 和 Windows。产品以“简单、可控、本地优先”为核心，支持 Markdown、富文本、代码块、检索、笔记引用，并允许用户将 NAS 作为主要数据存储位置。

### 1.2 核心问题
现有笔记软件功能复杂、学习成本高、数据存储方式不够可控。用户需要一个更轻、更清晰、适合长期积累个人知识库的笔记工具。

### 1.3 产品目标
- 提供简洁、稳定、跨端一致的笔记编辑体验。
- 支持 Markdown 与富文本两种编辑习惯。
- 支持代码块编辑，适合技术学习、项目记录和知识沉淀。
- 支持 NAS 本地/局域网存储，让用户掌控数据。
- 支持全文检索、标签、笔记引用、反向链接，逐步形成个人知识库。
- 界面采用 macOS 风格，清爽、克制、易懂。

### 1.4 非目标
- v1 不做多人协作。
- v1 不做复杂项目管理、任务管理、日历系统。
- v1 不做公开发布站点。
- v1 不强依赖云服务。
- v1 不做复杂 AI 知识库自动化，AI 能力可作为后续增强。

## 2. 目标用户

### 2.1 主用户
独立开发者、学生、研究者、产品经理、内容创作者，以及希望管理长期知识资产的个人用户。

### 2.2 用户特征
- 有大量碎片化资料、学习笔记、代码片段、项目文档。
- 希望数据长期可迁移、可备份、可掌控。
- 不想被复杂功能打扰。
- 可接受基础设置，但希望日常使用足够顺手。

## 3. 核心场景

### 3.1 快速记录
用户打开软件，快速新建一篇笔记，写下想法、资料摘录或会议记录。

### 3.2 技术学习
用户在笔记中混合编写文字、Markdown 标题、列表、代码块、链接和图片。

### 3.3 构建知识库
用户通过文件夹、标签、双向链接和反向链接，把零散笔记逐渐组织成主题知识库。

### 3.4 检索旧知识
用户输入关键词，快速找到相关笔记、标题、正文片段、标签和引用关系。

### 3.5 跨设备访问
用户在 Mac 上整理笔记，在 iPhone 上查看或快速补充，在 Windows 上继续编辑。

## 4. 功能需求

### 4.1 笔记编辑
#### Markdown 编辑
- 支持 Markdown 语法输入。
- 支持实时预览或编辑/预览分屏。
- 支持标题、列表、引用、表格、链接、图片、任务列表。
- 支持快捷键和 Markdown 自动补全。

#### 富文本编辑
- 支持所见即所得编辑。
- 支持标题、加粗、斜体、下划线、删除线、列表、引用、链接、图片、表格。
- Markdown 与富文本之间应尽量保持结构一致。

#### 代码块编辑
- 支持 fenced code block。
- 支持语言选择。
- 支持语法高亮。
- 支持复制代码。
- 支持代码块内等宽字体、缩进、换行设置。

### 4.2 知识库组织
#### 文件夹
- 支持多层文件夹。
- 支持拖拽移动笔记。
- 支持固定常用文件夹。

#### 标签
- 支持为笔记添加多个标签。
- 支持标签筛选。
- 支持标签重命名与合并。

#### 笔记引用
- 支持 `[[笔记标题]]` 形式引用。
- 输入 `[[` 时弹出笔记选择器。
- 支持未创建笔记引用，点击后创建。
- 支持反向链接列表。

#### 知识库视图
v1 建议提供三种轻量视图：
- 文件视图：按文件夹组织。
- 标签视图：按主题聚合。
- 关系视图：展示当前笔记的出链、反链和相关笔记。

图谱视图可作为 v1.5 或 v2 功能，不作为首版核心依赖。

### 4.3 检索
- 支持标题、正文、标签全文检索。
- 支持最近打开、最近修改。
- 支持按文件夹、标签、日期过滤。
- 搜索结果展示命中的标题、路径和正文摘要。
- 支持快捷搜索入口。

### 4.4 数据存储与同步
#### 本地优先
- 笔记以开放格式保存，优先考虑 Markdown 文件加元数据。
- 附件以独立文件夹存储。
- 索引数据库可本地生成，允许重建。

#### NAS 存储
- 支持选择 NAS 挂载目录作为知识库位置。
- 当前用户 NAS 为绿联云 NAS，首版按“系统已挂载目录”方式接入，不在 App 内实现 NAS 账号登录。
- 支持 SMB/WebDAV/iCloud Drive/本地文件夹等可访问路径，首版优先支持“已挂载目录”。
- 支持检测路径可用性。
- 支持离线缓存与冲突提示，冲突自动覆盖不作为默认行为。

#### 数据结构建议
```text
KnowledgeBase/
  notes/
    inbox/
    projects/
    topics/
  assets/
    images/
    files/
  .notebase/
    index.db
    settings.json
```

补充约定：
- `notes/` 下的 `.md` 文件是真实源数据。
- 富文本是编辑层能力，落盘优先统一回 Markdown。
- 图片与附件不内嵌为主存储，独立放在 `assets/images/` 与 `assets/files/`。
- `.notebase/` 保存索引、缓存和 App 元数据，可重建，不作为真实内容源。
- 不同格式内容通过索引关系、相对路径引用和笔记元数据进行关联。

### 4.5 跨端支持
#### macOS
- 首要平台。
- 原生 macOS 风格三栏布局。
- 支持快捷键、窗口拖拽、系统深浅色。

#### iPhone
- 以阅读、检索、快速记录为主。
- 编辑能力完整但界面简化。
- 支持最近笔记、收件箱、搜索。

#### Windows
- 功能与 macOS 保持一致。
- 视觉风格保留简洁，但适配 Windows 字体与窗口习惯。

## 5. 用户体验原则

- 简洁优先：默认只展示高频功能。
- 内容优先：编辑区始终是视觉中心。
- 本地可控：用户知道数据在哪里。
- 渐进复杂：知识库能力逐步显露，不强迫用户先学习方法论。
- 跨端一致：核心概念一致，界面按设备优化。

## 6. MVP 范围

### 6.1 MVP 必做
- macOS 桌面端基础版本。
- 新建、编辑、删除、移动笔记。
- Markdown 编辑与预览。
- 富文本基础编辑。
- 代码块高亮。
- 文件夹管理。
- 标签。
- 全文检索。
- `[[笔记引用]]` 与反向链接。
- 本地文件夹/NAS 挂载目录作为知识库位置。

### 6.2 MVP 可延后
- iPhone 完整编辑体验。
- Windows 深度适配。
- 图谱视图。
- AI 自动总结、自动分类。
- 插件系统。
- 端到端加密。

## 7. 技术栈决策

### 7.1 确认技术栈
- 桌面端：Tauri 2 + React + TypeScript。
- UI 层：Tailwind CSS + shadcn/ui 作为基础组件层，按 macOS 风格二次定制。
- 富文本编辑器：TipTap。
- Markdown 处理：remark/unified。
- 代码块编辑：CodeMirror 6。
- 代码高亮：Shiki。
- 本地数据库与检索：SQLite + FTS5。
- 文件系统：Tauri filesystem 插件与原生能力结合。
- 状态管理：优先使用 React 内建状态，必要时引入 Zustand。
- 移动端策略：iPhone 首版不同步开发，桌面端稳定后再评估 React Native/Expo。

### 7.2 决策理由
- 选择 Tauri 而不是 Electron，是为了更轻的安装体积和更贴近桌面文件系统的本地优先体验。
- 选择 TipTap，是因为它适合承载富文本与结构化文档模型，后续更容易兼容 Markdown、引用和知识库语义。
- 选择 SQLite FTS5，是因为本地全文检索成熟、稳定、易调试，足够覆盖个人知识库场景。
- 选择“绿联云 NAS 挂载目录”而不是内置 SMB/WebDAV 登录，是为了先做稳可用的 MVP，降低同步和鉴权复杂度。
- 选择先桌面后移动，是为了先把最难的编辑器、文件模型和知识库关系能力打牢。

## 8. 风险与对策

| 风险 | 影响 | 对策 |
| --- | --- | --- |
| Markdown 与富文本转换复杂 | 数据丢失或格式错乱 | 统一内部文档模型，明确不支持的格式 |
| NAS 同步冲突 | 笔记被覆盖 | 使用修改时间、内容哈希和冲突副本 |
| 跨端成本过高 | 进度拖慢 | 首先完成桌面端 MVP |
| 知识库功能过重 | 用户学习成本上升 | 先做引用、反链、标签，图谱后置 |
| 编辑器实现复杂 | 核心体验不稳定 | 采用成熟编辑器框架 |
