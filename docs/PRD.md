# 个人知识库笔记软件 PRD v0.1

## 1. 产品概述

### 1.1 产品定位
一款面向个人使用的跨端笔记与知识库软件，覆盖 macOS、iPhone 和 Windows。产品以“简单、可控、本地优先”为核心，支持 Markdown、代码块、检索、笔记引用，并允许用户把 NAS / WebDAV 作为可选同步目标，而不是主存储入口。

### 1.2 核心问题
现有笔记软件功能复杂、学习成本高、数据存储方式不够可控。用户需要一个更轻、更清晰、适合长期积累个人知识库的笔记工具。

### 1.3 产品目标
- 提供简洁、稳定、跨端一致的笔记编辑体验。
- 支持 Markdown-first 的稳定编辑体验。
- 支持代码块编辑，适合技术学习、项目记录和知识沉淀。
- 支持 NAS 本地/局域网存储，让用户掌控数据。
- 支持全文检索、标签、笔记引用、反向链接，逐步形成个人知识库。
- 界面采用 macOS 风格，清爽、克制、易懂。

### 1.4 非目标
- v1 不做多人协作。
- v1 不做复杂项目管理、任务管理、日历系统。
- v1 不做公开发布站点。
- v1 不强依赖云服务。
- v1 不做复杂 AI 知识库自动化，AI 能力可作为后续增强。

## 2. 目标用户

### 2.1 主用户
独立开发者、学生、研究者、产品经理、内容创作者，以及希望管理长期知识资产的个人用户。

### 2.2 用户特征
- 有大量碎片化资料、学习笔记、代码片段、项目文档。
- 希望数据长期可迁移、可备份、可掌控。
- 不想被复杂功能打扰。
- 可接受基础设置，但希望日常使用足够顺手。

## 3. 核心场景

### 3.1 快速记录
用户打开软件，快速新建一篇笔记，写下想法、资料摘录或会议记录。

### 3.2 技术学习
用户在笔记中混合编写文字、Markdown 标题、列表、代码块、链接和图片。

### 3.3 构建知识库
用户通过文档类型、笔记本、标签、双向链接和反向链接，把零散笔记逐渐组织成主题知识库。

### 3.4 检索旧知识
用户输入关键词，快速找到相关笔记、标题、正文片段、标签和引用关系。

### 3.5 跨设备访问
用户在 Mac 上整理笔记，在 iPhone 上查看或快速补充，在 Windows 上继续编辑。

## 4. 功能需求

### 4.1 笔记编辑
#### Markdown 编辑
- 支持 Markdown 语法输入。
- 支持实时预览或编辑/预览分屏。
- 支持标题、列表、引用、表格、链接、图片、任务列表。
- 支持快捷键和 Markdown 自动补全。

- 首版编辑以 Markdown-first 为准。
- 后续可以补充更强的富文本层，但不能破坏 Markdown 文件作为真实源数据的原则。

#### 代码块编辑
- 支持 fenced code block。
- 支持语言选择。
- 支持语法高亮。
- 支持复制代码。
- 支持代码块内等宽字体、缩进、换行设置。

### 4.2 知识库组织
#### 文档类型与笔记本
- 每篇文档必须包含两个独立属性：
  - `documentType`：表示文档类型，首版至少支持 `note`、`todo`、`journal`
  - `notebook`：表示是否归属于某个笔记本；未归类时为空
- `Notes / Todo / Journal` 视图只展示未归类到 notebook 的文档。
- 当用户把文档移动到某个 notebook 后，该文档应从原来的主类型列表中移除，并显示在目标 notebook 下。
- 当用户把文档从 notebook 中移出后，该文档应回到其 `documentType` 对应的主类型列表。
- `documentType` 用于表达文档语义，`notebook` 用于表达归档位置，两者不能互相替代。

#### 笔记本
- 支持用户创建多个 notebook。
- notebook 是文档的归档位置，而不是额外标签。
- 支持把文档移动到 notebook，或从 notebook 移回未归档主列表。
- notebook 下展示该 notebook 内的实际文档。

#### 标签
- 支持为笔记添加多个标签。
- 支持标签筛选。
- 支持标签重命名与合并。

#### 笔记引用
- 支持 `[[笔记标题]]` 形式引用。
- 输入 `[[` 时弹出笔记选择器。
- 支持未创建笔记引用，点击后创建。
- 支持反向链接列表。

#### 知识库视图
v1 建议提供三种轻量视图：
- 主列表视图：按 `documentType` 展示未归档文档。
- notebook 视图：展示归属于某个 notebook 的文档。
- 关系视图：展示当前笔记的出链、反链和相关笔记。
- 图谱视图：P1 基础版展示真实 note 节点、tag 节点、wikilink 边和 tag 边。
- 图谱视图优先做轻量 2D 邻居网络，不在 P1 引入复杂布局算法。

### 4.3 检索
- 支持标题、正文、路径、标签全文检索。
- 支持最近打开、最近修改。
- 支持按文档类型、notebook、标签、日期过滤。
- 搜索结果展示命中的标题、路径和正文摘要。
- 支持快捷搜索入口。
- P1 基础版通过 `Cmd/Ctrl + K` 命令面板承载本地全文搜索和筛选。

### 4.4 数据存储与同步
#### 本地优先
- 笔记以开放格式保存，优先考虑 Markdown 文件加元数据。
- 附件以独立文件夹存储。
- 索引数据库可本地生成，允许重建。

#### 本地离线知识库
- App 默认提供本地离线知识库路径。
- 首版默认路径为 macOS `~/Documents/NoteBase`。
- App 启动时优先扫描本地离线路径，而不是远端目录。
- 如果本地路径为空，自动初始化知识库目录结构。

#### NAS / WebDAV 同步
- NAS 不再作为首屏主存储入口，而是作为可选同步目标。
- 用户未启用同步时，App 只依赖本地离线知识库工作。
- 用户启用同步后，才需要填写 WebDAV / NAS 配置。
- 同步入口放在主界面右上角，未配置或异常时显示警示态。
- 启动时如果远端已配置且可通信，App 对比本地与远端状态，再让用户决定同步方向。
- 首版同步以手动触发为主，冲突自动覆盖不作为默认行为。
- 首版同步保存本地 manifest 元数据，用于更稳的增量判断与冲突检测。
- 当检测到文件冲突时，先让用户按文件选择保留本地或远端版本。

#### 数据结构建议
```text
KnowledgeBase/
  notes/
    note/
    todo/
    journal/
    notebooks/
      Product/
      Research/
  assets/
    images/
    files/
  .notebase/
    index.db
    settings.json
```

补充约定：
- `notes/` 下的 `.md` 文件是真实源数据。
- 首版编辑器以 Markdown-first 为准，后续富文本能力也必须回写为 Markdown。
- 图片与附件不内嵌为主存储，独立放在 `assets/images/` 与 `assets/files/`。
- 媒体库需要展示每个附件被哪些文档引用，并能识别当前未被任何文档引用的附件。
- 未引用附件支持在媒体库中直接删除；已被文档引用的附件在 P1 阶段不提供删除入口，避免误删正文依赖资源。
- 媒体库支持按时间、名称、大小排序，并为图片、视频、PDF、音频提供基础预览。
- 同步冲突需要在应用内展示逐文件列表，并允许用户按文件选择保留本地或保留远端版本。
- 命令面板搜索结果需要高亮命中词，并记住最近一次的类型 / notebook / tag 筛选条件。
- 搜索排序需要优先考虑标题精确 / 前缀命中，其次标签和路径命中，并对最近修改文档给予轻量加权。
- 媒体库需要支持对当前筛选结果做批量选择，并批量删除其中未被引用的附件。
- `.notebase/` 保存索引、缓存和 App 元数据，可重建，不作为真实内容源。
- 不同格式内容通过索引关系、相对路径引用和笔记元数据进行关联。
- 未归类文档按 `documentType` 存放在主类型目录下，例如 `notes/note/`、`notes/todo/`、`notes/journal/`。
- 已归类到 notebook 的文档应物理移动到 `notes/notebooks/<Notebook Name>/` 下，而不是仅更新元数据。
- 文档移动到 notebook 时，必须同步维护其内部相对附件路径，避免图片与文件引用失效。
- 早期测试库中可能仍然存在 `inbox / projects / topics` 等历史目录；当前版本在加载知识库时自动迁移 markdown 文件：
  - `inbox` -> `notes/note/`
  - `projects` -> `notes/notebooks/Projects/`
  - `topics` -> `notes/notebooks/Topics/`
- 迁移时需要补齐 `documentType` / `notebook` 元数据，并同步维护正文中的相对附件路径。
- 如果启动时发生旧目录迁移，界面应提示用户迁移了多少文档以及迁移来源。
- 迁移历史需要写入 `.notebase/migration-log.json`，设置页可回看最近迁移记录。
- 空目录或空 notebook 应直接提供“在这里新建”入口，降低首次使用阻力。

### 4.5 跨端支持
#### macOS
- 首要平台。
- 原生 macOS 风格、内容优先的连续工作台布局。
- 支持快捷键、窗口拖拽、系统深浅色。

#### iPhone
- 以阅读、检索、快速记录为主。
- 编辑能力完整但界面简化。
- 支持最近笔记、未归档文档、搜索。

#### Windows
- 功能与 macOS 保持一致。
- 视觉风格保留简洁，但适配 Windows 字体与窗口习惯。

## 5. 用户体验原则

- 简洁优先：默认只展示高频功能。
- 内容优先：编辑区始终是视觉中心。
- 本地可控：用户知道数据在哪里。
- 渐进复杂：知识库能力逐步显露，不强迫用户先学习方法论。
- 跨端一致：核心概念一致，界面按设备优化。

## 6. MVP 范围

### 6.1 MVP 必做
- macOS 桌面端基础版本。
- 新建、编辑、删除、移动笔记。
- Markdown 编辑与预览。
- 标题 + 正文分离的 Markdown 编辑。
- 代码块插入与预览。
- notebook 管理。
- 标签。
- 全文检索。
- `[[笔记引用]]` 与反向链接。
- 默认本地离线路径作为知识库位置。
- 可选远端 WebDAV / NAS 同步入口。

### 6.2 MVP 可延后
- iPhone 完整编辑体验。
- Windows 深度适配。
- 图谱视图。
- AI 自动总结、自动分类。
- 插件系统。
- 端到端加密。

## 7. 技术栈决策

### 7.1 确认技术栈
- 桌面端：Tauri 2 + React + TypeScript。
- UI 层：Tailwind CSS + shadcn/ui 作为基础组件层，按 macOS 风格二次定制。
- 富文本编辑器：TipTap（后续增强方向，当前主编辑链路仍以 Markdown-first 为准）。
- Markdown 处理：remark/unified。
- 代码块编辑：CodeMirror 6。
- 代码高亮：Shiki。
- 本地数据库与检索：SQLite + FTS5。
- 文件系统：Tauri filesystem 插件与原生能力结合。
- 状态管理：优先使用 React 内建状态，必要时引入 Zustand。
- 移动端策略：iPhone 首版不同步开发，桌面端稳定后再评估 React Native/Expo。

### 7.2 决策理由
- 选择 Tauri 而不是 Electron，是为了更轻的安装体积和更贴近桌面文件系统的本地优先体验。
- 选择 TipTap，是因为它适合承载未来更强的结构化编辑层，但当前版本必须先保证 Markdown-first 编辑稳定。
- 选择 SQLite FTS5，是因为本地全文检索成熟、稳定、易调试，足够覆盖个人知识库场景。
- 选择“绿联云 NAS 挂载目录”而不是内置 SMB/WebDAV 登录，是为了先做稳可用的 MVP，降低同步和鉴权复杂度。
- 选择先桌面后移动，是为了先把最难的编辑器、文件模型和知识库关系能力打牢。

## 8. 风险与对策

| 风险 | 影响 | 对策 |
| --- | --- | --- |
| Markdown 与富文本转换复杂 | 数据丢失或格式错乱 | 统一内部文档模型，明确不支持的格式 |
| NAS 同步冲突 | 笔记被覆盖 | 使用修改时间、内容哈希和冲突副本 |
| 跨端成本过高 | 进度拖慢 | 首先完成桌面端 MVP |
| 知识库功能过重 | 用户学习成本上升 | 先做引用、反链、标签，图谱后置 |
| 编辑器实现复杂 | 核心体验不稳定 | 采用成熟编辑器框架 |
